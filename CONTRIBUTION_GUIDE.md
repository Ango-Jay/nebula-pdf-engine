# NEBULA PDF ENGINE

## Conventions to follow while writing code
## 1. File Naming Conventions

- File names should be in kebab-case. e.g. `file-name.md` or `file-name.js` or `file-name.css`

- File names should be in lowercase.

- File names should be descriptive. e.g. `user-profile.md` or `user-profile.js` or `user-profile.css`

## 2. Folder Naming Conventions

- Folder names should be in kebab-case. e.g. `folder-name` or `folder-name`

- Folder names should be in lowercase.

- Folder names should be descriptive. e.g. `user-profile` or `user-profile`

## 3. Variable Naming Conventions

- Variable names should be in camelCase. e.g. `variableName`

- Variable names should be descriptive. e.g. `userName` or `userProfile`

- Check the spelling of the variable name to be sure its correct.

## 4. Function Naming Conventions

- Function names should be in camelCase. e.g. `functionName`

- Function names should be descriptive. e.g. `getUser` or `getUserProfile`

- Check the spelling of the function name to be sure its correct.

## 5. Commenting Conventions

- Comments should be used to explain code logic..

- Comments should be used to explain complex code.

- Comments should be used to explain code that is not self-explanatory.

- Comments should be used to explain code that is not easy to understand.

## 6. Code Indentation Conventions

- Code should be indented using 2 spaces.

- Code should be indented properly to make it readable.

- Code should be indented properly to make it easy to understand.

- Use prettier to format code.

## 7. General Conventions

- Avoid using short names for variables, functions, and files.

- Avoid using short names for folders.

- Avoid using short names for anything that is not a comment.

- Avoid using short names for anything that is not a comment.

## 8. Pull Request Conventions

- Pull requests should be descriptive.

- Pull requests should be detailed.

- Pull requests should be clear.

- Pull requests should be concise.

- Pull requests should be easy to understand.

- Pull requests should be easy to review.

- Pull requests should be easy to merge.

- Pull requests should be easy to test.

- Pull requests should be easy to deploy.

- Pull requests should be easy to manage.
